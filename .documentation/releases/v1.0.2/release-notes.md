## Bug Fix

### CSP blocking the harness in Development environments

**Symptom:** After upgrading to v1.0.1, the harness at `/api-test-spark/` appeared blank
or empty when the consuming app ran in `Development` mode (e.g. in Visual Studio or
`dotnet run`). The harness worked correctly in v1.0.0.

**Root cause:** The `Content-Security-Policy` header set on every `index.html` SPA fallback
response contained a fixed `connect-src` directive that only allowed specific HTTPS domains.
ASP.NET Core's Browser Link and hot-reload tooling inject JavaScript that opens WebSocket
connections back to localhost (`ws://localhost:{port}`) — the CSP blocked those connections,
preventing the SPA from initialising.

The bug was present in v1.0.0 too but not observed because the SPA was simpler and Browser
Link interference was less likely to prevent render. The v1.0.1 SPA changes (additional lazy
chunks, renderMarkdown, API Doc Builder) increased initialisation complexity, making the
CSP-blocked WebSocket connections more reliably cause a blank screen.

**Fix:** `ApiTestSparkExtensions.cs` — when serving `index.html`, detect
`IHostEnvironment.IsDevelopment()` and append localhost origins to `connect-src`:

```
ws://localhost:* ws://127.0.0.1:* http://localhost:* http://127.0.0.1:*
```

In non-Development environments the CSP is unchanged.
