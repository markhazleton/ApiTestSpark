## Highlights

v1.4.0 promotes the remote API experience from one configured target to a full **Remote API Profiles** model. Host apps can seed multiple named remote APIs from `Program.cs`, and users can personalize the list from the browser Config page without rebuilding the host app.

Each profile has a stable id, display name, description, base URL, OpenAPI URL, credentials, and profile-scoped headers. The home screen, Remote API Explorer, and Remote API Doc Builder now render profile-specific entries so users can move between external APIs without overwriting saved settings.

The release also tightens the security boundary introduced in v1.3.0. Server-configured profile secrets are redacted from `/api-test-spark/config` and used only by `GET /api-test-spark/remote-spec?profileId=...`. Browser-created profiles are local browser configuration and fetch their OpenAPI documents directly from the browser, so they cannot drive arbitrary server-side URL fetches.

## New Features

### Remote API Profiles

Configure multiple remote APIs with `RemoteApiProfiles` in `Program.cs`, including name, description, base URL, OpenAPI URL, credentials, and default headers.

**Spec**: [View archived spec](specs/001-remote-api-list/spec.md)

### Browser Profile Management

The Config page can add, edit, delete, hide, reset, and persist browser-managed remote profiles. New browser profiles receive GUID ids. Visible profile names must be unique, and duplicate names are blocked before save with a clear validation alert.

### Profile-Specific Navigation and Docs

Each visible profile gets its own remote explorer and documentation entry. Generated remote documentation uses the selected profile's safe name and description instead of URLs or credential-bearing fields.

### Credential-Redacted Server Profiles

Server-provided API key values and bearer tokens are not serialized to the browser config endpoint. The proxy resolves server profiles by id and uses server-held credentials only.

## Compatibility

No breaking changes. The new `RemoteApiProfiles` collection and `RemoteApiProfile` model are additive. Existing single-remote options remain supported and seed one compatibility profile when no profile collection is configured.

## Upgrade Guide

Update the NuGet package:

```bash
dotnet add package ApiTestSpark --version 1.4.0
```

To configure multiple remote APIs, add profiles in `Program.cs`:

```csharp
app.MapApiTestSpark(options =>
{
    options.RemoteApiProfiles.Add(new RemoteApiProfile
    {
        Id = "orders-api",
        Name = "Orders API",
        Description = "Order management endpoints.",
        RemoteBaseUrl = "https://orders.example.com",
        RemoteOpenApiUrl = "https://orders.example.com/openapi.json",
        RemoteOpenApiApiKeyHeader = "x-api-key",
        RemoteOpenApiApiKeyValue = builder.Configuration["Orders:ApiKey"],
    });
});
```
