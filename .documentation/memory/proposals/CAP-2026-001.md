## Current State

**Principle VI — Observability & Logging (current text, relevant excerpt):**

> - `console.error` MAY only appear inside `catch` blocks for unrecoverable errors (MUST)
> - All API request/response/error/metric data MUST be routed through `useDebugStore` (MUST)

The current principle correctly bans `console.log` and limits `console.error`, but:
1. "Unrecoverable error" is undefined — audit finding QUAL1 shows a reasonable developer treating a recoverable catch (audit log failure) as "unrecoverable" and passing review.
2. The routing rule ("All API request/response/error/metric data") is scoped to API data — it is ambiguous whether storage, configuration, and component-layer diagnostics also require debug-store routing. Audit findings OBS1–OBS3 all involve non-API observability that falls in this gap.

## Proposed Change

### Principle VI — Observability & Logging (proposed additions, no removals)

Add the following clarifications after the existing bullet list:

**Define "unrecoverable":**

> An error is **unrecoverable** only when the current operation cannot complete and there is no meaningful fallback: e.g., a fatal render error in `ErrorBoundary`, an assembly load failure in the .NET layer, or a network error where no retry is possible. Secondary operation failures (audit log write, telemetry flush, build-info load) are **recoverable** — the primary operation succeeded. `console.error` MUST NOT be used for recoverable failures.

**Expand the routing rule scope:**

> All diagnostic output from `src/` — not only API calls, but also storage failures, configuration warnings, version mismatches, and component-level errors — MUST be routed through `useDebugStore.addError()` using the appropriate error category (`'Network' | 'API' | 'Configuration' | 'Unknown'`). The rule "All API request/response/error/metric data" is clarified to mean **all observable events within `src/`**, regardless of layer.

**Recoverable-failure guidance:**

> For truly non-critical failures where the user does not need to know (e.g., missing optional build metadata), silently returning `null` or a safe default is preferred over either console output or debug-store noise.

**Rationale**: The existing rule was sound but underspecified. Developers working in the utility and component layers interpreted the "API data" scope narrowly and filled the gap with console calls. Defining "unrecoverable" eliminates judgment ambiguity. Expanding the routing scope to all of `src/` closes the gap.

**No new MUSTs are added** — existing behaviour is clarified, not tightened.
