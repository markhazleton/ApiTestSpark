# ADR-006: Shared `buildCurl` Utility Extraction

## Status

Accepted

## Context

The response renderer needed a "Copy as cURL" button in the response panel, generating the same cURL format already produced by the request panel in `DebugPanel.tsx`. The existing `buildCurl` logic lived inline in `DebugPanel.tsx` with no reuse path.

## Decision

Extract `buildCurl({ method, url, headers, body })` into `src/utils/curlBuilder.ts`, re-export from `src/utils/index.ts`, and update `DebugPanel.tsx` to import from the shared utility. `EndpointTester.tsx` also imports from the same location.

Inlining a second copy was rejected (divergence risk). Passing a callback prop was rejected (over-engineering a pure function). The `src/utils/` layer is the correct Constitution §III home for a pure, stateless function with no React or store dependencies.

## Consequences

### Positive

- Single authoritative cURL format — any future change is made once
- Constitution §III layer separation satisfied: pure utility in `src/utils/`
- `DebugPanel.tsx` and `EndpointTester.tsx` share identical output format

### Negative

- One additional file in `src/utils/`; any future signature change must be coordinated across both callers
