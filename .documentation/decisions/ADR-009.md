# ADR-009: Remote API Profiles with Server-Profile-Only Proxying

## Status

Accepted

## Context

ApiTestSpark v1.3.0 introduced a single remote API target and a server-side OpenAPI spec proxy. The next product need was to support multiple remote APIs at once, with shared defaults configured in `Program.cs` and user-personalized profiles managed in the browser.

The key security constraint is that browser-managed profiles must not cause arbitrary server-side URL fetches. At the same time, server-configured profile credentials must remain useful for teams and demos without being serialized to `/api-test-spark/config`.

## Decision

Remote API configuration is represented as a list of `RemoteApiProfile` objects. Each profile has a stable id, display name, description, base URL, OpenAPI URL, credentials, and profile-scoped default headers.

Server-provided profiles are the only profiles eligible for the server-side spec proxy. The browser calls `GET /api-test-spark/remote-spec?profileId=...`; the server resolves the id against startup-configured profiles and uses only the server-held URL and credentials. Browser-created profiles are stored locally and fetch their OpenAPI documents directly from the browser.

Server-provided API key values and bearer tokens are redacted from `/api-test-spark/config`. The config response exposes profile metadata and configured flags, not secret values. Legacy single-remote options remain supported as an additive compatibility seed when no profile list is configured.

## Consequences

### Positive

- Multiple remote APIs can appear in navigation, explorer, and documentation flows without overwriting one global target.
- Server-held credentials remain off the browser config payload and are only used by id-bound server profiles.
- Browser-created profiles cannot submit arbitrary OpenAPI URLs to the server proxy.
- Existing single-remote consumers retain a compatibility path without a breaking change.

### Negative

- Browser-created profiles depend on the remote API allowing browser-side CORS access to its OpenAPI document.
- User-personalized browser profiles are local to the current browser storage and are not shared across devices or users.
- Profile ids become part of route and proxy behavior, so host-provided ids should be stable across deployments.
