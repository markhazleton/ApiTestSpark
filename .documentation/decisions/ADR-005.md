# ADR-005: Session-Only UI Preference in Existing Harness Store

## Status

Accepted

## Context

The response renderer needed to persist a pretty/minified JSON view preference across API calls within the same browser session. The preference must survive React re-renders triggered by new `data` props, but must reset on page reload (no `localStorage` persistence).

## Decision

Add `jsonViewMode: 'pretty' | 'minified'` and `setJsonViewMode` action to the existing `useHarnessConfigStore`. This store is already the canonical home for session-only UI state (Constitution §V), carries no `persist` middleware, and resets naturally on page reload.

Local `useState` was rejected because `ResponseView` remounts with each new `data` prop, resetting the preference on every API call. A new `useResponsePrefsStore` was rejected to avoid one-concern-per-store proliferation for a preference that naturally belongs under harness configuration.

## Consequences

### Positive

- Zero new stores — no barrel export changes, no new `create()`
- Constitution §V one-concern rule satisfied: view mode preference is a harness session concern
- Reset-on-reload behaviour is free (no persist middleware to strip)

### Negative

- `useHarnessConfigStore` grows slightly in scope; future session preferences should be evaluated against whether they warrant their own store
