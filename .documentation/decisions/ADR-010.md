# ADR-010: OAuth Token Configuration — Dual Browser/Server Acquisition with Secret Redaction

## Status

Accepted

## Context

Testers exercising APIs that require an OAuth2 bearer token had no in-tool way to obtain one — they had to acquire a token through an external process and paste the raw string into a Remote API profile's static Bearer Token field. That token had no defined origin, was never refreshed, and there was no way to exercise both a machine-identity flow (`client_credentials`) and a user-identity flow (`password` grant) against the same API from within the harness.

A server-side variant of the same problem existed for `Program.cs`-configured profiles: teams wanted to point a profile at an OAuth-protected API without ever putting a client secret in the browser, mirroring the existing `RemoteOpenApiBearerToken`/`RemoteOpenApiApiKeyValue` pattern where server-held credentials are used by the server's own proxy and never serialized to `/api-test-spark/config`.

A critical risk surfaced during design and was confirmed in adversarial review (`gates/critic.md` critic-001, SHOWSTOPPER): the harness's shared debug-capture pipeline forwards every request body to the debug panel and, when enabled, Application Insights. An OAuth token request's body contains `client_secret`/`password` in plaintext (required by the RFC 6749 form-urlencoded grant format) — without explicit handling, every token acquisition would leak the tester's credentials into telemetry.

## Decision

Two independent OAuth acquisition paths were built, sharing the "server holds secrets, browser never sees them" design already established for static bearer tokens:

**Browser-side** — a per-Environment (`localhost`/`test`/`other`) OAuth configuration (token endpoint, application client credentials, optional separate user-flow client credentials, test user credentials) with two distinct acquisition entry points: `acquireOAuthToken` (always fetches fresh — used by the explicit "Get Token" buttons, guaranteeing a silent-overwrite contract) and `ensureOAuthToken` (returns a cached, still-valid token when available — used only by automatic pre-fire callers, and hardcoded to `client_credentials` so a cleared or expired `password`-grant token is never silently re-acquired using stored test-user credentials). All token requests flow through the same shared `executeRequest` pipeline as every other API call (Constitution IV — no raw `fetch`), but a redaction wrapper substitutes `"***redacted***"` for `client_secret`/`password` values in the copy of the request body passed to debug callbacks, while the real, unredacted body still reaches the token endpoint. Remote API profiles opt in via `remoteUseOAuthToken`; if a valid token cannot be obtained, the request is blocked with a distinct error rather than being sent unauthenticated or silently falling back to a static token.

**Server-side** — a new `RemoteApiProfileOAuth` (`TokenEndpointUrl`, `ClientId`, `ClientSecret`) on `RemoteApiProfile.OAuth`, `client_credentials` only. The server acquires and caches a token in an in-memory, process-wide `ConcurrentDictionary` keyed by profile id (30-second expiry buffer), injecting `Authorization: Bearer <token>` when fetching the remote OpenAPI document and, when `EnableRemoteCallProxy` is enabled, when proxying endpoint calls. A failed acquisition logs a warning and proceeds without the header (fail-open, matching the existing `RemoteOpenApiBearerToken` behavior) rather than throwing. `/api-test-spark/config` exposes only a `remoteOAuthConfigured: true/false` boolean — the token endpoint URL, client id, and client secret are never serialized to the browser.

## Consequences

### Positive

- Testers can exercise both `client_credentials` and `password` grant flows against the same target API without ever pasting a raw token string.
- The critic-001 SHOWSTOPPER secret-leak risk is closed structurally (a redaction wrapper at the client boundary) rather than by convention, so it cannot regress silently as new callers are added.
- The `ensureOAuthToken`/`acquireOAuthToken` split makes "always fresh" vs. "reuse if valid" an explicit, type-level choice at each call site rather than an implicit behavior that differs by caller.
- Server-side OAuth reuses the exact "server holds secret, exposed only via a `*Configured` boolean, only works through the server proxy" pattern already established for static bearer tokens — no new security model to reason about.
- Both paths were verified against a live/mock OAuth2 provider before release, which caught and fixed an unrelated but serious defect: the acquired token was not being attached to the real outgoing request in one code path, only to a UI preview (see `.documentation/specs/001-oauth-token-config/plan.md` Implementation Notes and `pr-review/pr-7.md`).

### Negative

- Password grant support is provider-dependent — many identity providers (Entra ID, Okta, Auth0, Google) disable the Resource Owner Password Credentials grant by default per OAuth 2.1 guidance; this is documented as a known limitation, not a defect.
- The server-side OAuth token cache is a single static dictionary shared across every `MapApiTestSpark()` registration in the same process, keyed only by profile id. This is consistent with the existing `RemoteOpenApiBearerToken` proxy's process-wide characteristics, but a host registering the harness multiple times with colliding profile ids could see token cross-talk (informational finding L-01, PR #7 review; no known caller does this today).
- Server-side OAuth is `client_credentials` only — password grant was intentionally not extended server-side, since a `Program.cs`-configured profile represents a fixed application identity, not a rotating test user.
