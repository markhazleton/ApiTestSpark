# ADR-002: Dual-Build Strategy via VITE_BASE_PATH Environment Variable

## Status

Accepted

## Context

The React SPA previously deployed as a standalone Azure Static Web Apps site at `/`. Packaging it as a NuGet-embedded resource requires all asset references and client-side routes to resolve under `/api-test-spark/`. Both deployment targets must remain functional from the same source tree.

## Decision

Drive Vite's `base` config from `process.env.VITE_BASE_PATH ?? '/'`. The `BrowserRouter` reads `import.meta.env.BASE_URL` as its `basename`. The `pack.ps1` script sets `VITE_BASE_PATH=/api-test-spark/` before running `npm run build`; the standalone SWA build leaves the env var unset, defaulting to `/`.

## Consequences

### Positive

- A single source tree produces both deployment targets with no branching or patching
- The standalone SWA deployment (Azure Static Web Apps CI) is entirely unaffected by the NuGet packaging work
- `pack.ps1` is the single canonical entry point for producing the NuGet package, preventing manual mistakes

### Negative

- `VITE_BASE_PATH` must be set and cleared correctly by any build script that calls `npm run build`; forgetting it produces an incorrectly rooted NuGet package with no build-time error
